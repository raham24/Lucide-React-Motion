import { Heart, Send } from "lucide-react-motion";

Every rendered icon carries a `data-motion-state` attribute on its `<svg>`. It reflects the current phase of the draw animation and lets you style the **surrounding UI** in lockstep with what the stroke is doing.

```html
<svg data-motion-state="resting"  …>…</svg>
<svg data-motion-state="drawing"  …>…</svg>
<svg data-motion-state="complete" …>…</svg>
```

No props, no React state, no callbacks. Just CSS selectors against the attribute.

---

## The three states

| State      | Meaning                                                                  |
| ---------- | ------------------------------------------------------------------------ |
| `resting`  | Idle. No animation is playing.                                           |
| `drawing`  | The stroke draw (the `active` variant) is currently animating.           |
| `complete` | The stroke draw finished and the icon is held at its fully-drawn state.  |

The initial server-rendered value is always `resting`, so SSR and hydration are safe.

---

## Why it exists

When you pair `trigger="parent-hover"` with a host color transition, the host's `hover:text-primary` releases the instant the cursor leaves — but with `onLeave="complete"` (the default) the stroke is still drawing. The result, before this attribute existed, was a visible color snap-back to the resting color mid-animation.

With `data-motion-state` you can keep the host color pinned for as long as the icon is doing anything other than resting:

<Preview>
  <div
    data-motion-icon-group
    className="inline-flex h-24 w-24 items-center justify-center rounded-md border border-fd-border text-fd-foreground transition-colors hover:text-primary has-data-[motion-state=drawing]:text-primary has-data-[motion-state=complete]:text-primary"
  >
    <Heart size={36} trigger="parent-hover" />
  </div>
</Preview>

```tsx
<div
  data-motion-icon-group
  className="
    text-foreground transition-colors
    hover:text-primary
    has-data-[motion-state=drawing]:text-primary
    has-data-[motion-state=complete]:text-primary
  "
>
  <Heart trigger="parent-hover" />
</div>
```

Hover, then quickly move away — the stroke color now holds through the full draw instead of snapping back.

## How transitions work

The simplest way to think about it: the state follows what the **stroke** is doing, with one extra rule for hover.

- The stroke isn't moving → `resting`
- The stroke is drawing right now → `drawing`
- The stroke is done drawing and the icon is being held there → `complete`

The "held there" part is what differs by trigger.

### Hover and parent-hover

`complete` is only held **while the cursor is over the icon (or its parent)**. Leave, and it goes back to `resting`.

Three scenarios cover everything:

**You hover, the draw finishes, you keep hovering, then you leave:**

```
  hover          draw           you           you
  enters         finishes       wait          leave
    │              │              │             │
    ▼              ▼              ▼             ▼
 resting  ──▶  drawing   ──▶  complete  ──▶  resting
```

**You hover and leave before the draw finishes:**

```
  hover          you           draw
  enters         leave         finishes
    │              │              │
    ▼              ▼              ▼
 resting  ──▶  drawing  ─ ─ ─ ▶  resting
              (keeps           (skips
               drawing)         "complete")
```

The state stays `drawing` after you leave so any host CSS that's pinned to the drawing state keeps its color until the stroke is actually done. Then it returns straight to `resting` — never lands on `complete`.

**You hover, the draw finishes, then you leave:**

```
 resting ──▶ drawing ──▶ complete ──▶ resting
                                       ▲
                                  the moment you
                                  leave the icon
```

That's it. `onLeave` (`complete` / `snap` / `redraw`) controls **what the animation does** when you leave, but the state always lands at `resting` once you're gone and the stroke has settled.

### Mount, in-view, click, manual

These triggers have no "leave," so `complete` just stays put — the icon is finished and there's nothing telling it to go back to rest.

```
 resting ──▶ drawing ──▶ complete   (and it stays here)
```

It only moves off `complete` when the trigger fires again:

- `click` — next click restarts the cycle.
- `mount` — never fires again, so it stays `complete` forever.
- `manual` — until you call `play()` (restart) or `reset()` (back to `resting`).
- `in-view` — scrolling out of view sends it back to `resting`, and scrolling back in restarts the cycle.

### Two edge cases worth knowing

- **Reduced motion.** No animation runs, so the state stays `resting`. CSS rules that key off `drawing` or `complete` simply never match.
- **`repeat={Infinity}`.** The draw never reports as finished, so the state stays `drawing` for the entire loop. `complete` is never reached.

## Tailwind v4 syntax

The canonical class shape is `has-data-[motion-state=…]:…` on the parent, or `data-[motion-state=…]:…` directly on the icon:

```tsx
// On the parent (sync host style with child icon state)
<div data-motion-icon-group className="has-data-[motion-state=drawing]:text-primary">
  <Heart trigger="parent-hover" />
</div>

// On the icon itself
<Heart
  trigger="hover"
  className="data-[motion-state=drawing]:scale-110 transition-transform"
/>
```

If you're not using Tailwind, the equivalent CSS is:

```css
[data-motion-icon-group]:has([data-motion-state="drawing"]),
[data-motion-icon-group]:has([data-motion-state="complete"]) {
  color: var(--primary);
}
```

`:has()` is supported in every modern evergreen browser.

## When to use this vs other patterns

- **You want host color / transform sync with the draw.** → `data-motion-state` selectors. No JS overhead, composes cleanly.
- **You need to run an imperative side effect when the draw begins or ends** (analytics, audio, chained animations). → This isn't covered yet. Lifecycle callback props (`onAnimationStart` / `onAnimationEnd`) are a possible future addition; for now you can read the attribute via a `MutationObserver` on the icon's `<svg>` if you really need it.
- **You want a different draw entirely.** → See [Custom motion](/docs/custom-motion). Custom variants still must use the `rest` / `active` label names if you want the state machine to track them correctly.

---

## Type

The attribute's value type is also exported from the package:

```ts
import type { MotionState } from "lucide-react-motion";
//          ^ "resting" | "drawing" | "complete"
```

You'll rarely need it — the attribute is read via CSS, not JS — but it's there for components that want to type a `MutationObserver` payload or a state mirror in their own component tree.
